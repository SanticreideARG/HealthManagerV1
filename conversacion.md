La joya a reciclar: el anti-overbooking
Es el activo central y se traduce casi 1:1. El hotel bloquea solapamiento por habitación en granularidad día (daterange). Vos necesitás bloquear por profesional en granularidad hora (tstzrange):
sqlCREATE EXTENSION IF NOT EXISTS btree_gist;

ALTER TABLE turnos
  ADD CONSTRAINT turnos_sin_solapamiento
  EXCLUDE USING gist (
    profesional_id WITH =,
    tstzrange(inicio, fin, '[)') WITH &&
  ) WHERE (estado <> 'cancelado');
El [) te sigue sirviendo igual: un turno que termina 10:00 y otro que arranca 10:00 no chocan. Y el patrón de la API se recicla entero: el CTE atómico (alta de paciente + turno en una sola sentencia, porque el driver neon-http no tiene transacciones interactivas) y la traducción de 23P01 → HTTP 409.
Ojo con una distinción crítica: el EXCLUDE solo previene solapamiento entre turnos; no valida que el turno caiga dentro de una ventana de trabajo. Eso es regla de negocio a nivel app (chequear contra la ventana antes de insertar). El constraint es el backstop contra race conditions, no la única validación.

El concepto nuevo central: ventanas de trabajo

ventanas_recurrentes  → profesional_id, dia_semana, hora_inicio, hora_fin,
                        duracion_turno, vigencia_desde, vigencia_hasta, activa
ventanas_excepciones  → profesional_id, fecha, tipo('agrega'|'bloquea'),
                        hora_inicio, hora_fin, motivo
Disponibilidad (reemplaza /disponibilidad) = expandir ventanas recurrentes de la fecha → aplicar excepciones → restar turnos no cancelados → trocear en slots de duracion_turno. Es pura aritmética de rangos; te la escribo cuando definamos duraciones. Descarté el modelo de "slots pre-materializados" porque es rígido para duraciones variables y sobreturnos.

Sobreturnos. ¿Algún profesional necesita agendar 2 pacientes en el mismo horario a propósito? Si sí, el EXCLUDE duro los bloquea y hay que modelarlos aparte (cambia el constraint). Si no, mantenemos el bloqueo estricto.

Tus 7 decisiones → estructura de datos
1. Sobreturnos (staff sí, paciente no). El constraint gana una condición: es_sobreturno boolean. Los turnos normales nunca se pisan; un sobreturno queda fuera del EXCLUDE a propósito:
sqlEXCLUDE USING gist (profesional_id WITH =, tstzrange(inicio, fin, '[)') WITH &&)
  WHERE (estado <> 'cancelado' AND NOT es_sobreturno);
El flujo online no puede setear es_sobreturno (solo profesional/administrativo por API con rol) → el paciente jamás genera un solapamiento. Exactamente lo que pediste.
2. Duración por ventana con presets — ¿complica la BD? No. Y es importante que entiendas por qué: el turno guarda inicio/fin concretos (timestamps), y el constraint opera sobre el rango real, agnóstico a cómo lo segmentaste. La duración (20/30/40/60, editable) vive en ventanas.duracion_turno (con profesionales.duracion_turno_default como fallback) y solo afecta el cálculo de slots ofrecidos — capa de aplicación, no DDL. Corolario deseable: si mañana cambiás la duración de una ventana, los turnos ya agendados conservan su rango original. Cero migraciones, cero riesgo en la BD. La segmentación es un problema de UI/disponibilidad, nunca de integridad.
3. Multidisciplina + filtro. profesionales.especialidad (o disciplina) → filtro en la landing (lo dejé como héroe en el manual). El consultorio no es un segundo eje de anti-overbooking (lo descartaste bien): el recurso reservable es el profesional. Multi-profesional, mono-sede.
4. Modo de confirmación flexible. ventanas.modo_confirmacion enum('automatico','aprobacion'), con default heredable de profesional/config. Turno online hereda el modo de la ventana: automatico → confirmado, aprobacion → solicitado. Staff siempre crea confirmado. Cubre "todas las posibilidades según necesidad".
5. Obras sociales. Catálogo obras_sociales + M2M profesional_obras_sociales. En pacientes: obra_social_id + nro_afiliado. Si la OS del paciente no está en la lista del profesional → turno marcado particular + aviso de costo (no bloqueante, con conformidad). Sin procesamiento de pago.
6. Sede. Campo ubicacion informativo (en profesional o ventana), se muestra al reservar. Sin multi-tenant en Fase 1.
Estados finales del turno: solicitado → confirmado → en_sala → atendido, más ausente, cancelado y bloqueo (agenda sin paciente).
7. Recordatorios serverless (Fase 3) — buena noticia
Tu instinto de evitar Evolution API es correcto para serverless. Meta discontinuó la API on-premise el 23 de octubre de 2025; desde entonces el WhatsApp Cloud API es el único camino para integraciones nuevas. Y lo clave: el Cloud API corre enteramente en servidores de Meta — sin infraestructura self-hosted, sin aprovisionamiento, sin dependencias. Son llamadas HTTP (Graph API) desde una función de Vercel; encaja perfecto con tu arquitectura. Evolution API, en cambio, se despliega con Docker manteniendo una sesión activa de WhatsApp Web — justo el container que no querés montar. Message Central + 2
El costo a tener en cuenta: desde julio de 2025 Meta cobra por mensaje (per-message), las conversaciones de servicio iniciadas por el cliente en la ventana de 24hs son gratis, pero los mensajes de plantilla iniciados por el negocio se cobran por envío. Un recordatorio de turno es plantilla categoría utility (más barata que marketing). Para mail serverless: Resend o Amazon SES, ambos HTTP puro. Ozonetel
